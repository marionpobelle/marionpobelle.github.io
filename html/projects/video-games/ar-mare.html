OVERVIEW

Set in 1920 off the coasts of Brittany, Ar-Mare follows Morgan Le Duienn, a fisher-woman turned light-keeper. After the Great War, which spurs her to escape France’s new policies, she takes up the job at the Ar-Mare lighthouse only to immediately find out that the place is not what it seemsー and thus begins the nightmare. The game tells a story of loneliness and of one-sided, abusive love, and ownership.

This game experience is a proof of concept developed as part of our second-year master's program at ENJMIN over six months. It is therefore far from perfect, but it includes all the key elements that make up this beautiful and scary experience.
This build contains 2 days of gameplay, with 3 tasks each day and 2 main puzzles.

My duties included:

  • Develop gameplay features (player controller, interactions…)
  • Implement User Interface (design menus, set up navigation…)
  • Asset integration (User Interface, 3D models, animations…)
  • Improve the game designer’s work environment with homemade tools (Google Sheet)
  • Performance control and optimization
  • Collaborating closely with environment artists, sound designers and game designers
  • Maintain a feature list with regular updates so that everyone can track progress
  • Writing technical documentation (risk assessment sheet, performance budget sheet…)
  • And more!

As one of the primary point of contact for the rest of the team, I created and maintained the game’s feature list and documentation. After that, I detailed the structure of each feature on paper, prototyped them, processed the team’s feedback as I went, and then worked on and owned them until polished.

Some of my teammates wanted to learn scripting, so I did my best to help them, be available and allow them to implement small features themselves.

TASK & EVENT SYSTEM

One of the main challenges of Ar Mare was the designing and application process of the level design, as a linear game with in-game events occurring at a specific time / context. The goal was to make the iteration pipeline the simplest possible for the designers and to have a solid way of managing the different events and what triggers them.

Google sheet was used as an external tool to manage the data of the events. It dictates which trigger will activate which receiver. Moreover it can apply some modifiers to the way an event occurs such as order, how many triggers are needed, or how many times the event can be triggered.

The game’s first day events.

Each day’s event sheet has an object sheet linked to it to see all objects present in the corresponding map.  

To implement the triggers/receiver logic, we used the actor components of Unreal which let us add them to whatever actor we want. And thus implement their behavior to its owner actor. We also use them to detect the triggering and receiving actors of the scene, and send it to the Google sheet. 

Trigger and receiver actors.

The event trigger implements a simple “Trigger” and “Untrigger” function, to be called when the trigger actor is activated. The event receiver implements a delegate “OnReceiverActivated” called when the event is activated.

To retrieve the data from the Google sheet, we use a Http request to download the .tsv and then parse the info into the events. 

        
public void ALDManager::SetLdEventsActors()
{
    _ldEvents = new TArray();
    TArray<FLDEventData*> eventsData;
    _eventsTable->GetAllRows<FLDEventData> (TEXT("EventData"), eventsData);
    // create new events for the play (to avoid changes on the model)
    for (FLDEventData* ldEvent : eventsData)
    {
        int i = _ldEvents->Add(new FLDEventData(*ldEvent));
        FLDEventData* newEvent = (*_ldEvents)[i];
        // get the triggers and receivers actors with their names
        newEvent->Triggers = RetrieveAllObjectsWithComponentInNameArray<UEventTrigger>(ldEvent->TriggersName);
        newEvent->ActivatedReceivers = RetrieveAllObjectsWithComponentInNameArray<UEventReceiver>(ldEvent->ActivatedReceiversName);
        _eventLogic = GetGameInstance()->GetSubsystem<UEventLogicSubSystem>();
        _eventLogic->SetupEventTracking(_ldEvents);
        UE_LOG(LogArMare, Warning, TEXT("Event Actors Set"));
    }
}

Setting the Actors for the Google Sheet events.

After getting the information in the event data we search for each actor linked to an event to add its name to the list of the event. We have to do this because Unreal has a system of names and labels. Labels are the actors’ names we see and names are the actors’ true names. The difficulty is that the label names (that we use to display actors in the Google sheet) are not loaded to the game’s build. So we have to get each actor’s true name and save them in the data before we play or build the game.

Then, when we play the game, we iterate through the saved names (and not labels) to link the actors to the events. 

        
for (int i = 2; i < Lines.Num(); i++)

TECHNICAL DOCUMENTATION

I was responsible for writing some technical documentation to ensure a proper workflow throughout the project's allocated time. I mainly focused on writing a risk assessment sheet and a performance budget sheet.

The risk assessment sheet encompassed every discipline within the team and focused on prevention over post-resolution. The procedure was as follows:
The first step was to identify a problem, its consequences, and which category it needed to be affected in. Then, I computed its impact on development as well as its probability of happening. Finally, I established a prevention strategy as well as a main prevention action. As an example:

  • Problem: Recording the voice acting too early or too late.
  • Consequences: Potentially missing audio assets or not enough time to work on flagging or processing.
  • Category: Audio.
  • Impact: High.
  • Probability: 50%.
  • Prevention Strategy: Think early about a limit date for recording the voice actor.ress that takes into account the voice processing, subtitle flagging and integration.
  • Prevention Action: Find the voice actor.ress in advance, find out their availability and set the final list for voice lines not too early to allow time for changes but not too late for the recording sessions.
